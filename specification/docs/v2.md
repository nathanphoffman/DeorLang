# V2 Roadmap

## Function Annotations

`[test]`, `[deprecated]`, `[pure]` were removed from v1 ŌĆö decorators are not part of the language. V2 will introduce an alternative mechanism for marking test functions, flagging deprecations, and signaling pure functions. The exact syntax is TBD but will not use `[...]` bracket annotations.

---

Features explicitly deferred from v1. These are design decisions made but implementation postponed.

---

## No `while` Loop ŌĆö Closed

`while` is not planned for v2. Bounded iteration belongs in `for` + `break`; truly unbounded iteration belongs in `rust` blocks alongside the low-level machinery it always requires. See [Loops ŌĆö No `while` Loop](loops.md#no-while-loop--by-design) for the full reasoning.

---

## No Native `dict` Type ŌĆö Closed

A native `dict` / `HashMap` type is not planned for v2. A dictionary requires two generic type parameters (`K` and `V`) which conflicts with Deor's one-type-per-generic constraint. More importantly, performance-sensitive lookups belong at the Rust level where you choose the right data structure ŌĆö `HashMap`, `BTreeMap`, `IndexMap`, etc. Use a `rust` block to build the structure and a `raw` variable to carry it. See [Rust Interop ŌĆö The `raw` Type](interop.md#the-raw-type).

---

## `throw` with Struct Support

Currently `throw` accepts strings only. V2 will allow throwing any struct ŌĆö useful for structured error types:

```
throw Error("message", context)    # v2
```

For now, use a `func` shape parameter for structured error handling, or bind a string and `throw` it for hard stops.

---

## String Performance (`&str` Inference)

Currently all strings are owned `String` in Rust and concatenation always uses `format!()`. V2 will introduce smart `&str`/`String` inference ŌĆö the transpiler will use borrows where provably safe to reduce allocations in string-heavy code. No syntax changes required; this is purely a transpiler optimization.

---

## Literal Predicate Validation

Currently `Positive val = -1` produces a runtime `None` (predicate fails silently). V2 will evaluate predicates on known literals at transpile time and emit a compile error:

```
Positive val = -1    # v2: transpiler error, not runtime None
```

---


## Hex and Binary Numeric Literals

Currently only decimal integers with optional underscore separators are supported. V2 will add:

```
int mask = 0xFF
int flags = 0b1010_1100
```

For v1, use a `rust` block if hex or binary literals are required.

---

## `throw` Catchability

Currently `throw` is always unrecoverable (`panic!()`). V2 may introduce a recoverable throw pattern if a `Result`-like concept is added to the language.

---

## `assert` Built-in

V2 will add `assert` as a built-in for use in test functions. Takes a `bool` expression and an optional string message. Exact test-function syntax TBD ŌĆö see Function Annotations above.

```
assert result is expected
assert result is expected, "factorial(5) should be 120"
```

In v1, write assertions using `if` and `throw`, or use a `rust` block with `assert_eq!` directly.
